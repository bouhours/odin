# Security Policy

## Supported Versions

Security updates are generally provided for the latest stable release. Older versions may receive fixes at the maintainers' discretion.

| Version | Supported |
| ------- | --------- |
| Latest stable release | Yes |
| Older releases | Best effort |
| Unsupported / end-of-life releases | No |

## Reporting a Vulnerability

Please do not report security vulnerabilities through public GitHub issues.

If you believe you have found a security vulnerability, please report it privately by using one of the following options:

- GitHub private vulnerability reporting, if enabled for this repository.
- Email: `md.gestion.projet@free.fr`.

Please include as much information as possible:

- A clear description of the vulnerability.
- Steps to reproduce the issue.
- Affected versions or commits.
- Potential impact.
- Proof of concept, if available.
- Suggested mitigation, if known.

## Response Expectations

Maintainers will make a best effort to:

1. Acknowledge receipt of the report.
2. Investigate and validate the issue.
3. Work on a fix or mitigation when appropriate.
4. Coordinate disclosure with the reporter.
5. Publish an advisory or release notes if needed.

Please allow reasonable time for investigation before public disclosure.

## Coordinated Disclosure

We ask reporters to avoid public disclosure until maintainers have had time to investigate and release a fix or mitigation.

## Security Updates

Security fixes may be released as patch versions, advisories, or documented mitigations depending on the impact and project release process.
